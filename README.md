The Gaming Room is a company that wanted to expand their existing game, Draw It or Lose It, into a web-based application that could support multiple teams and players at the same time. Their main goal was to move from a single-platform setup to something more scalable and accessible across different devices. The software needed to handle multiple game sessions, manage users and teams, and make sure everything ran smoothly without conflicts or duplication.

One thing I think I did really well in this project was clearly breaking down the system requirements and connecting them to actual design decisions. Instead of just listing what the system should do, I focused on explaining how and why certain design choices were made, like using a singleton pattern to control game instances or structuring the system to support scalability. That made the document feel more practical and not just theoretical.

Working through the design document actually made the coding part a lot easier. Having everything planned out ahead of time, like class structure, system architecture, and constraints, meant I was not guessing while writing code. It gave me a roadmap, so instead of constantly backtracking, I could focus on implementing what I had already designed.

If I could go back and revise one part, I would probably improve the section on system constraints and platform evaluation. I touched on different operating systems and environments, but I think I could go deeper into why one platform would be a better choice over another, especially in terms of performance and long-term scalability.

When it came to interpreting user needs, I focused on what the client actually cared about, like being able to run multiple games at once, keeping the system responsive, and making it easy for users to join and play. I translated those needs into design features like efficient memory use, unique identifiers for games and teams, and a structure that supports concurrent users. Considering user needs is important because if the system does not solve the user’s problem or is frustrating to use, then it does not really matter how well it is built.

My approach to designing the software was pretty structured. I started by understanding the requirements, then moved into modeling the system with diagrams and outlining the architecture before thinking about code. In the future, I would keep using that same approach, but I would spend more time comparing different design patterns and technologies upfront so I can make more informed decisions before development.
